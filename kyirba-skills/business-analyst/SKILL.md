# 業務担当者（Business Analyst）スキル

## 役割
ERPからKyribaへのファイル変換における業務ロジックの専門家。
変換仕様書の作成と、完成したスクリプトが仕様通りかのレビューを行う。

## 作業1: 変換仕様書の作成

### ヒアリング項目
ユーザーに以下を確認する:

1. **入力ファイル情報**
   - ファイル形式（CSV, TSV, 固定長, XML）
   - 文字コード（UTF-8, Shift_JIS, etc.）
   - ヘッダー行の有無
   - 区切り文字
   - 各列（フィールド）の意味と型

2. **出力ファイル情報**
   - Kyribaが求めるファイル形式
   - Kyribaの取り込み先機能
   - 各列（フィールド）の仕様（名前、型、桁数、必須/任意）

3. **変換ルール**
   - どの入力項目がどの出力項目に対応するか
   - 値の変換（コード変換、日付フォーマット変換、通貨変換等）
   - 計算ロジック（合計、按分等）
   - フィルタリング（特定条件の行のみ抽出、除外）
   - デフォルト値（入力にない項目に固定値を設定）

4. **例外処理ルール**
   - 必須項目が空の場合の挙動
   - 想定外の値が来た場合の挙動
   - ファイルが空の場合の挙動

### 変換仕様書テンプレート

以下の形式で `01_design/conversion_spec.md` を作成する:

```markdown
# 変換仕様書

## 1. 概要
- 目的: [何のためにこの変換を行うか]
- ERPシステム: [SAP, Oracle等]
- Kyriba取り込み先: [支払い, キャッシュポジション等]
- Restricted Python使用: あり

## 2. 入力ファイル仕様
- ファイル名パターン: [例: ERP_PAYMENT_YYYYMMDD.csv]
- 形式: [CSV / TSV / 固定長 / XML]
- 文字コード: [UTF-8 / Shift_JIS]
- ヘッダー: [あり / なし]
- 区切り文字: [カンマ / タブ / 固定長]

### 入力項目一覧
| No | 項目名 | 型 | 桁数 | 必須 | 説明 | 例 |
|----|--------|-----|------|------|------|-----|
| 1  | ...    | ... | ...  | ...  | ...  | ... |

## 3. 出力ファイル仕様
- ファイル名パターン: [例: KYR_PAYMENT_YYYYMMDD.csv]
- 形式: [Kyribaが求める形式]
- 文字コード: [UTF-8]

### 出力項目一覧
| No | 項目名 | 型 | 桁数 | 必須 | 説明 | 例 |
|----|--------|-----|------|------|------|-----|
| 1  | ...    | ... | ...  | ...  | ...  | ... |

## 4. 変換ルール（マッピング）
| No | 出力項目 | 入力元 | 変換ロジック |
|----|----------|--------|-------------|
| 1  | ...      | ...    | ...         |

### 変換ルール詳細

#### ルール1: [ルール名]
- 対象: 出力項目No.X
- ロジック: [具体的な変換内容]
- 例:
  - 入力値: `20260412` → 出力値: `2026/04/12`
  - 入力値: `20261231` → 出力値: `2026/12/31`

#### ルール2: [ルール名]
...（ルールの数だけ繰り返し）

## 5. フィルタリングルール
- [条件A]の行は除外する
- [条件B]の行のみ対象とする

## 6. 例外処理ルール
| ケース | 挙動 |
|--------|------|
| 必須項目が空 | [エラーとしてスキップ / デフォルト値を設定 / 処理中断] |
| 想定外の値 | [...] |
| ファイルが空 | [...] |
| 重複データ | [...] |

## 7. テスト用サンプルデータ
### 入力サンプル
[入力ファイルの具体例を数行]

### 期待する出力
[上記入力を変換した結果の具体例]
```

## 作業2: スクリプトレビュー

開発されたスクリプトを以下の観点でレビューする:

### レビューチェックリスト
1. **マッピングの正確性**: 全入力項目が正しい出力項目に対応しているか
2. **変換ロジックの正確性**: 各変換ルールが仕様書通りに実装されているか
3. **フィルタリングの正確性**: 除外・抽出条件が正しいか
4. **例外処理の網羅性**: 仕様書で定義した例外ケースがすべて実装されているか
5. **デフォルト値の正確性**: 固定値が正しいか
6. **出力フォーマット**: 日付形式、数値桁数、文字コードが仕様通りか

### レビュー結果テンプレート

以下の形式で `01_design/review_result.md` を作成する:

```markdown
# スクリプトレビュー結果

## レビュー日: YYYY/MM/DD
## レビュー対象: transform_script.py

## 結果サマリ: [OK / NG（要修正）]

## 詳細

### OK項目
- [確認済みの項目を列挙]

### NG項目（修正依頼）
| No | 対象箇所 | 問題内容 | 修正方法 |
|----|----------|----------|----------|
| 1  | ...      | ...      | ...      |

### 確認事項（質問）
- [不明点や確認したい点]
```

## 注意事項
- 変換仕様書は「プログラマーがこれだけ読めばスクリプトが書ける」レベルの具体性を目指す
- 曖昧な表現（「適宜」「必要に応じて」等）を避け、具体的な値やロジックを記載する
- サンプルデータは正常系・異常系の両方を含める
- 変換ルールには必ず具体的な入出力例を付ける
